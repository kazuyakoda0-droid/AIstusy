Principles

分解の3原則

「タスクの粒度をどこで切るか」を決める3つの基準。これに反したら、まだ分解が足りていないサイン。

Principle 01

1タスク = 1判断

1つのタスクに、判断ポイントを複数持たせない。"内容を確認して、判断して、登録する" は3タスクに分ける。

Principle 02

入力と出力を明示

各タスクで「何を受け取って／何を返すか」を JSON や箇条書きで明文化する。曖昧な引き継ぎは事故の元。

Principle 03

"検証可能" にする

タスクの出力を、人間が後から検証できる粒度にする。"なぜそう判断したか" のログが残ること。

Checklist

分解の "やり過ぎ／足りなさ" を見抜くチェックリスト

分解は「細かければよい」わけではない。実用上の最小単位を狙う。

✓
そのタスクの "成功" を一文で書けるか？
"何が起きたら成功か" を一文で書けないなら、まだ複合タスクの可能性が高い。
✓
前のタスクの出力 = 次のタスクの入力、になっているか？
前タスクが「商品名と数量」を返し、次タスクが「商品名と数量」を受け取る、というように噛み合わせる。
✓
人間が "判断" すべきポイントが、AIに紛れていないか？
"高額契約の最終承認" のような重要判断は、AIに任せず Human タスクとして明示する。
✓
そのタスクは独立してテストできるか？
"このタスクだけ" 渡されて、AIが正しく動けるか試せるか？ できないなら、依存関係が複雑すぎる。
✓
タスク数が多すぎないか？（10〜15が目安）
100タスクのパイプラインは現実的に保守できない。"集約して1タスク" にできるか再検討する。
分解は "AI への翻訳作業"。
曖昧な依頼を、機械が動ける粒度の指示に変える。

タスク分解 Task Decomposition

大きな業務を、AIが処理できる小さな単一タスクに切り分ける作業。料理を "下処理 → 切る → 炒める → 盛り付ける" に分けるのと同じ。各工程は単純で、検証も容易。

スキーマ Schema

タスクの入力／出力の "形" を定義したもの。"会社名は文字列、納期は日付、金額は数値" のように決めておく。注文票のフォーマット。決まった欄に決まった種類の情報を入れることで、次工程が処理しやすくなる。

単一責任 Single Responsibility

1タスクには1つの責務だけ持たせる原則。"複数のことをやろうとする" タスクは分割する。"営業もしつつ経理もやる" 人は仕事の質が落ちる。専門に分けたほうが結局速い。

人間タスク Human Task

パイプラインの中で "人間が担う" と明示されたタスク。判断・承認・例外処理が中心。"課長承認が必要" "金額が100万円超で部長判断" など、人間がハンコを押す箇所。

粒度 Granularity

タスクの "細かさ"。粗すぎると AI が処理できず、細かすぎると保守できなくなる。レシピが "煮る" だけだと曖昧、"鍋を 30 秒揺すって、塩を 1g 振る" だと細かすぎる。中間が良い粒度。

業務フロー図 Process Flow

タスクの順番・分岐・人間の介入点を 1 枚にまとめた図。設計の出発点。"病院の外来フロー" のような、誰が・いつ・何をするかを描いた業務マップ。
