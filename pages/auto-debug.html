Session 07 ／ コーディングエージェント

自律デバッグ
AIが、自分のエラーを自分で直す。

プログラミングで最も時間がかかるのは「エラー解析と修正」。コーディングエージェントは、実行 → エラー検知 → 原因推定 → 修正のループを自分で回し、完成まで導く。

Concept
In one sentence

エラーは、"次の手がかり"。
AIはそれを読んで、自分で原因を辿り、自分で直す。

Diagram 01

4ステップのデバッグループ

これまで人間がやっていた「動かない → エラー文を読む → 原因を考える → 修正」を、AIが内部で繰り返す。

Step 01

実行

書いたコードを実行する。動けば次へ。動かなければエラー情報を取得。

Step 02

エラー検知

エラーメッセージ、行番号、スタックトレースを読み取る。

Step 04

再実行

修正を適用して再実行。成功するまで繰り返す。

Step 03

原因推定＆修正

エラー文から原因を推定。コードを修正、ライブラリを追加、設定を変更。

Diagram 02

実際のデバッグループ

「PDFから表を抽出する」スクリプトを作るときの、エージェント内部の動き。人間は一切介入しないのに、3回のループで完成する。

Round 1
スクリプトを作成して実行
Error
ImportError: No module named 'tabula'
Fix
tabula-py が未インストール。pip install tabula-py を実行する。
Round 2
再実行
Error
JavaNotFoundError: 'java' executable not found in PATH
Fix
tabulaにはJavaが必要。代替として pdfplumber に切り替える。
Round 3
pdfplumber でコードを書き直して再実行
Success
表データを CSV に出力完了。タスク終了。
プログラミングの最大の時間泥棒は "デバッグ"。
それをAIが代行できれば、開発速度は跳ね上がる。
Implication

非エンジニアにとっての意味

プログラミングの本質的な難しさは「動かないものを動かす」こと。これをAIが肩代わりすると、非エンジニアでも実用ツールを内製できる。

01

"何ができるか" の発想が広がる

「これは無理だろう」と諦めていた業務改善のアイデアが、現実的になる。Excelマクロでは難しかった処理も、Pythonスクリプト＋AIでサクッと実現できる。

02

"動けばよい" レベルの内製が増える

商用品質のシステムでなくとも、「自分の業務だけで使う小ツール」なら自律デバッグで十分な品質に到達できる。現場発のシャドーITが、健全な内製に転化する可能性。

03

注意：完璧ではない

複雑なバグ、設計レベルの問題、外部依存の不具合などはAIだけでは解決できない。「動いた＝正しい」とは限らないので、業務利用前のテストは人間の責任で。

Plain Language

用語解説 — むずかしい言葉をやさしく

このページに出てくる専門用語を、業務や日常に置き換えて理解する。

バグ Bug

プログラムの中の不具合・間違い。"レシピの中の誤字" のようなもの。それだけで料理が失敗する。

デバッグ Debug

バグを見つけて直す作業。プログラマーの仕事の大半。"なぜ料理が失敗したか" を、工程を遡って原因特定するのに似ている。

エラーメッセージ Error Message

プログラムが動かないときに出てくる、原因のヒント文。"塩がありません" のような警告文。AIはこれを読んで自分で材料を取りに行ける。

自律デバッグ Self-debugging

AIが出力結果を見て、自分でミスに気付いて直すこと。"煮込みが足りなかった" と気付いてもう5分火にかける、ベテランの調理感覚。

Analogy

専門用語のまま考えるとモヤモヤする概念も、身近な例に置き換えると一気に輪郭がつかめる。
