Phase 01 ／ AIの進化の系譜

ルールベース
条件分岐する電卓。

人間が事前に書いた「もしAならB」というルールに、忠実に従って動く時代のAI。融通は利かないが、決められたことは確実にこなす。

In one sentence

"if A then B" を積み重ねた、頭のいい電卓。

Diagram 01

仕組み — 決められた手順を粛々と実行する

人間がすべての分岐ルートをあらかじめ設計し、AIはその通りにYES/NOを判定して結論に至る。「考えている」ように見えても、条件分岐のツリーを辿っているだけ。

IF 件名に「請求書」を含む
→
「経理」フォルダへ
ELSE IF 送信元が顧客ドメイン
→
「顧客対応」フォルダへ
ELSE IF 件名に「会議」を含む
→
「予定」フォルダへ
ELSE — 受信トレイへ（人間が判断）
Pseudocode

本質は、こう書ける

どんなに複雑なルールベースシステムも、根本はこの「条件分岐の積み重ね」に過ぎない。

# メール振り分けの「AI」（ルールベース）
function classify(mail):
  if "請求書" in mail.subject:
    return "経理"
  elif mail.sender in customer_domains:
    return "顧客対応"
  elif "会議" in mail.subject:
    return "予定"
  else:
    return "未分類"
In practice

身の回りにある「ルールベース」

私たちは既に多くのルールベースシステムに囲まれている。重要なのは「これはAIか？」という問いではなく「どこまでが事前のルールで、どこからが学習か」という区別。

01

自動販売機

「投入金額 ≥ 商品価格 かつ ボタンが押された かつ 在庫がある」→ 商品を出す。完全にルールベース。例外（紙幣が斜めに入った等）は処理できず、エラーになる。

02

経費精算システムの自動承認

「金額 ＜ 1万円 かつ 勘定科目が定義済み」→ 自動承認、それ以外は上長へ。社内規程の電子化＝ルールベースAIの典型。

03

初期のチャットボット（電話の自動応答）

「1を押せば営業時間、2を押せばお問い合わせ…」。決められた選択肢の中でしか動けない。

ルールベースは "事前に書かれた人間の知恵" を実行するシステム。
AIが 「学ぶ」 前の時代のかたち。
Limits

限界 — なぜ次の時代が必要だったのか

ルールベースは堅牢だが、現実の業務は「ルールに収まらないこと」だらけ。ここから機械学習の必要性が生まれた。

Strength
得意なこと
  • 結果が常に同じ（再現性）
  • 「なぜそう判断したか」が明示できる（説明可能性）
  • 少ないデータでも動く
  • 規則が明確な業務（経理、社内規程）
Weakness
苦手なこと
  • 例外パターンに弱い（想定外＝動かない）
  • 曖昧さ・グラデーションが扱えない
  • ルールが数千個になると保守不能
  • 「この写真は猫か犬か」のような問題に手も足も出ない
