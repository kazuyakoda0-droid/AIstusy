エージェント間の "受け渡し" — スキーマで形を固定する

各エージェントの入力・出力を JSON のような構造化データで定義する。"受け渡しの型" さえ決まっていれば、中身は自由に差し替えられる。

// Agent 01（仕様書抽出）の Output ／ Agent 02 の Input
{
  "customer_name": "株式会社〇〇",
  "items": [
    {
      "product_code": "A-123",
      "quantity": 100,
      "inspection": "検査項目A"
    }
  ],
  "deadline": "2025-10-01",
  "total_amount": 1500000,
  "source_doc": "spec_v3.pdf"  // 追跡用
}

出力スキーマを固定すると、Agent 01 を差し替えても Agent 02 はそのまま動く。"AI部品" の差し替えが効くアーキテクチャになる。

Patterns

パイプラインの代表的トポロジー

業務の性質によって、エージェントの並べ方を変える。

Pattern 01

直列型 ／ Sequential

A → B → C と1本道で繋ぐ。各段で前段の出力を受けて加工する。最もシンプルで、デバッグしやすい。

A → B → C → 出力
Pattern 02

並列型 ／ Parallel

同じ入力から、複数のエージェントが同時に処理する。最後にマージ。時短に効く。

┌→ B1 →┐
A ┤       ├→ M → 出力
└→ B2 →┘
Pattern 03

分岐型 ／ Conditional

判定エージェントの結果で、次に呼ぶエージェントを変える。"金額 ≥ 100万なら法務へ" のような業務分岐に有効。

A → 判定 → 高額？
      Yes → 法務
      No → 通常
Pattern 04

ループ型 ／ Iterative

"出力が条件を満たすまで繰り返す" 型。修正提案 → 評価 → 修正提案、を品質OKまでループ。上限回数は必ず設定する。

A → B → 評価
  ↑    NG ↙
OK → 出力
パイプラインの強さは "配線の明確さ" で決まる。
AI自体の賢さより、繋ぎ方が結果を左右する。
Plain Language

用語解説 — むずかしい言葉をやさしく

このページに出てくる専門用語を、業務や日常に置き換えて理解する。

パイプライン Pipeline

複数のエージェントを順番に繋いだ処理の流れ。工場のベルトコンベア。1工程の出力が、次工程の入力になる連なり。

スキーマ Schema

エージェント間でやり取りされるデータの "型"。決められた項目・型のみを許す。"バトンの規格" を統一する。形が違うバトンは渡せない、を機械的に保証する。

直列／並列 Sequential / Parallel

エージェントを1本道で並べるか、横並びで同時実行するか。"1人ずつ報告" ⇔ "全員一斉に動き出す"。スピード重視なら並列、整合性重視なら直列。

条件分岐 Conditional Branching

判定の結果で、次に呼ぶエージェントを切り替える。"金額が100万円超なら部長承認、それ以下なら課長承認" の経路分け。

ループ／反復 Iterative Loop

"OKになるまで繰り返す" 構造。修正→評価を繰り返すパターンが代表的。レポートを書き直して上司レビューに通すサイクル。"OKもらえるまで何度でも" を、AIに任せる。

トレーサビリティ Traceability

パイプラインの各段で "何が、いつ、誰のせいで起きたか" を追跡可能にすること。宅配便の追跡番号。どの段階でどう動いたかを後から辿れる仕組み。
